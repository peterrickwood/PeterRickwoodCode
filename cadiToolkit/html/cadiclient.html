CADI Client User Guide

Assumptions: This guide assumes that you have sucessfully installed the CADI client. If you wish to use the client in distributed mode, you must have installed and started any CADI servers, and disabled any firewall on your computer. See the CADI Server user guide

Distributed and non-distributed modes

You can run the toolkit in 2 different modes, standalone and distributed.

Stand-alone mode

If you just want to run on your local machine, you launch the CADI toolkit by typing:

java -cp /tmp/cadiclient CadiClient /tmp/cadiclient

You will need to change /tmp/cadiclient to be the path to where you installed the cadi client.

Distributed mode

If you just want to do computations over several (or many) computers, you need to launch the toolkit by typing:

java -cp /tmp/cadiclient CadiClient /tmp/cadiclient --distributed

The toolkit will then have a look on the network to see if there are any servers available for computation.
$ java -cp /tmp/cadiclient CadiClient /tmp/cadiclient --distributed
....
scanning network for available nodes... please wait
waiting for replies from available nodes....
found 128 nodes

If it finds any servers, it will use them for any following computation. If it fails to find any available servers, it defaults to stand-alone mode.

Step-by-step Examples

The easiest way to gain an initial appreciation of what the toolkit can do for you is to go through the step-by-step examples.

Automatic code downloading

The CADI client has the neat feature that it automatically upgrades itself, if you tell it to. You dont need to do anything to obtain the latest bug-fixes or latest release of the software -- it gets downloaded automatically whenever you start up the CADI client, provided you answer YES to enable this feature during the install of the client, and provided you have a working network connection when you start the toolkit. If you have turned this option on, here is how it works:

Network class loading

When you install the CADI client, you actually dont download and install all the code that is required for the client to run. The small piece of code that you run when you run the client (CadiClient.class) is merely a small 'stub' program that is clever enough to establish a network connection and download the rest of the code it needs. Every time it starts, it attempts to download the cadi software. The toolkit stores the URLs that it tries to download software from in a file called repositoryURLs.txt. It tries each of the entries in this file, in order. If it cannot download the software from any entry in this file, and cannot find a local copy of the software, it prints an error message and exits.

Example -- always downloading the current version

As an example, if you always want to run the latest version of the CADI client, you can specify a URL like http://galitsin.anu.edu.au/caditk/softwarerepository/current/ during the install process, and then every time you start the client, it will download the cadi software from this URL.

Example -- always running version 0.1

When you install the toolkit, if you always want to run, say, version 0.1 of the toolkit, you can specify a URL that points to that version of the software (such as http://galitsin.anu.edu.au/caditk/softwarerepository/0.1/). Alternatively, you can add the line http://galitsin.anu.edu.au/caditk/softwarerepository/0.1/ as the top line in the repositoryURLs.txt file in the cadiclient directory.

Example -- disabling network class loading

When you install the cadi client, a copy of the CADI software is placed in the file caditk.jar. If you wish to disable network class loading, and always have the toolkit start up with the version of the CADI software 'frozen' at the time of download, you need to imake an entry like this the first line in your repositoryURLs.txt file:
file:/path/to/cadiclient/caditk.jar

Example -- adding additional code repositories in case one fails

If you open the file repositoryURLs.txt in the cadiclient directory, you will find that it has several entries. Let us assume that your file has the following entries:
http://galitsin.anu.edu.au/caditk/softwarerepository/current/
http://rses.anu.edu.au/~peterr/cadi/caditk/softwarerepository/current/
Suppose that you know of another URL where the CADI software is located. (say http://cadi.software.net/software/current/). You simply need to add this entry as a new line to the repositoryURLs.txt file, and the CADI client will look at this URL if it cannot find the code from any of the other listed URLs.

CADI client settings

There is a file .caditk (note the period), which specifies some environment variables for the toolkit. You should not need to mess with these, but you can take a look -- most of them should be self-explanatory. This file is created when you run the Cadi Installer.
